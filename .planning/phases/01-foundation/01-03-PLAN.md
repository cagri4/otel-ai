    1. Create `src/lib/validations/hotel.ts`:
       - `hotelSettingsSchema` as z.object:
         - name: z.string().min(1, 'Hotel name is required').max(100)
         - address: z.string().max(255).optional().or(z.literal(''))
         - city: z.string().max(100).optional().or(z.literal(''))
         - country: z.string().max(100).optional().or(z.literal(''))
         - timezone: z.string().refine() that validates the string is a valid IANA timezone using `Intl.DateTimeFormat(undefined, { timeZone: tz })` in a try/catch. Error message: 'Invalid timezone'
         - contactEmail: z.string().email('Invalid email').optional().or(z.literal(''))
         - contactPhone: z.string().max(30).optional().or(z.literal(''))
       - Export `HotelSettingsInput = z.infer<typeof hotelSettingsSchema>`

    2. Create `src/app/(dashboard)/settings/actions.ts`:
       - 'use server' directive
       - Import createClient from `@/lib/supabase/server`
       - Import hotelSettingsSchema from `@/lib/validations/hotel`
       - Export `async function updateHotelSettings(prevState: any, formData: FormData)`
         a. Create Supabase server client
         b. Verify user auth via `supabase.auth.getUser()` — return `{ error: 'Unauthorized' }` if no user
         c. Parse formData using hotelSettingsSchema.safeParse:
            - Extract fields from formData using Object.fromEntries
            - If validation fails, return `{ error: 'Validation failed', fieldErrors: parsed.error.flatten().fieldErrors }`
         d. Update hotels table:
            ```
            supabase.from('hotels').update({
              name: parsed.data.name,
              address: parsed.data.address || null,
              city: parsed.data.city || null,
              country: parsed.data.country || null,
              timezone: parsed.data.timezone,
              contact_email: parsed.data.contactEmail || null,
              contact_phone: parsed.data.contactPhone || null,
              updated_at: new Date().toISOString(),
            }).eq('id', hotelId)
            ```
            NOTE: The hotel_id comes from the JWT via RLS. The .eq('id', hotelId) is needed because RLS SELECT policies use the JWT hotel_id to match. Get hotelId by first querying the user's profile or by reading from the JWT. Simplest approach: query `supabase.from('hotels').select('id').single()` — RLS will return only the user's hotel.
         e. Call `revalidatePath('/settings')` and `revalidatePath('/')` to refresh cached data
         f. Return `{ success: true }` or `{ error: dbError.message }` on failure

    3. Create `src/components/forms/hotel-settings-form.tsx` ('use client'):
       - Props: `{ hotel: Hotel }` (Hotel type from database.ts)
       - Use react-hook-form with zodResolver(hotelSettingsSchema)
       - defaultValues populated from hotel prop (map snake_case DB fields to camelCase schema fields)
       - Use `useActionState` (React 19) with the updateHotelSettings Server Action
       - Fields (all using shadcn FormField/FormItem/FormLabel/FormControl/FormMessage):
         a. Hotel Name (Input, required)
         b. Address (Input or Textarea)
         c. City (Input)
         d. Country (Input)
         e. Timezone (TimezoneSelect from react-timezone-select):
            - value: field.value (the IANA string)
            - onChange: `(tz) => field.onChange(tz.value)` — CRITICAL: extract .value, not the whole object (see Pitfall 3 in research)
         f. Contact Email (Input, type="email")
         g. Contact Phone (Input, type="tel")
       - Submit button with loading state
       - Show success toast or message when save completes
       - Show field-level errors from Zod validation

       ANTI-PATTERN: Do NOT pass the entire timezone object to the form field. The `react-timezone-select` onChange returns `{ value: "Europe/Istanbul", label: "...", ... }`. You MUST extract `.value` only. If you pass the object, the DB will store "[object Object]".

       - Page title: "Hotel Settings"
       - Brief description: "Configure your hotel's basic information"

    5. Add a "Settings" link in the dashboard layout navigation (created in Plan 02) pointing to /settings.
