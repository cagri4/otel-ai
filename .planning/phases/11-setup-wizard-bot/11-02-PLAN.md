<task type="auto">
  <name>Task 1: Create wizard bot webhook handler</name>
  <files>src/app/api/telegram/wizard/route.ts</files>
  <action>
Create `src/app/api/telegram/wizard/route.ts` following the exact pattern of the existing employee bot handler at `src/app/api/telegram/[slug]/route.ts`, but adapted for the wizard bot:

1. Set `export const runtime = 'nodejs'` and `export const dynamic = 'force-dynamic'`

2. Import `after` from `next/server`, `TelegramUpdate` from `@/lib/telegram/types`, `handleWizardMessage` and `handleWizardCallback` from `@/lib/telegram/wizard/wizardActions`

3. Export `POST(req: Request): Promise<Response>`:
   - **Step 1: Validate webhook secret.** Read `x-telegram-bot-api-secret-token` header. Compare against `process.env.SETUP_WIZARD_WEBHOOK_SECRET`. If mismatch, return 403 "Forbidden". This is the ONLY non-200 response (same pattern as employee bot handler).
   - **Step 2: Parse body.** Cast `await req.json()` as `TelegramUpdate`. Extract `body.message` and `body.callback_query`.
   - **Step 3: Early exit for irrelevant updates.** If neither `message?.text` with `message?.chat?.id` NOR `callback_query?.id` with `callback_query?.data`, return 200 silently. This handles photos, stickers, etc.
   - **Step 4: Schedule async work via after() and return 200 immediately.** Inside `after(async () => { try { ... } catch (error) { console.error('[Wizard] Handler error:', error); } })`:
     - If `message?.text && message?.chat?.id`: call `await handleWizardMessage(message)`
     - Else if `callback_query?.id && callback_query?.data`: call `await handleWizardCallback(callback_query)`
   - **Step 5: Return 200.** `return new Response('', { status: 200 })`

4. Add comprehensive JSDoc header explaining:
   - Fixed route (not slug-based) because wizard is a single global bot
   - Handles both message and callback_query (unlike employee bots which only handle message)
   - Same after() pattern as employee bots to prevent Telegram retry storms
   - Env vars needed: SETUP_WIZARD_BOT_TOKEN, SETUP_WIZARD_WEBHOOK_SECRET

No rate limiting on the wizard route — it is accessed by the hotel owner only (not guest-facing), and the owner interacts at most 6 times per session.
  </action>

Create `src/app/api/admin/register-wizard-webhook/route.ts` — a one-time admin endpoint to register the wizard bot's webhook URL with the Telegram Bot API:

1. Set `export const runtime = 'nodejs'` and `export const dynamic = 'force-dynamic'`

2. Import `createServerClient` from `@/lib/supabase/server` and `cookies` from `next/headers`

3. Export `POST(req: Request): Promise<Response>`:
   - **Step 1: Auth guard.** Create a Supabase server client (using the same pattern as admin pages). Call `supabase.auth.getUser()`. If no user, return 401. Check `user.email === process.env.SUPER_ADMIN_EMAIL`. If not admin, return 403.
   - **Step 2: Read env vars.** Read `SETUP_WIZARD_BOT_TOKEN`, `SETUP_WIZARD_WEBHOOK_SECRET`, and `NEXT_PUBLIC_APP_URL`. If any are missing, return 400 with JSON error `{ error: 'Missing required env vars' }`.
   - **Step 3: Call Telegram setWebhook.** `fetch(`https://api.telegram.org/bot${botToken}/setWebhook`, { method: 'POST', headers: { 'Content-Type': 'application/json' }, body: JSON.stringify({ url: `${appUrl}/api/telegram/wizard`, secret_token: webhookSecret, drop_pending_updates: true, allowed_updates: ['message', 'callback_query'] }) })`
   - **Step 4: Check response.** Parse Telegram response as JSON. If `result.ok` is true, return 200 with `{ success: true, description: result.description }`. If false, return 500 with `{ error: result.description }`.

4. Add JSDoc explaining this is a one-time admin endpoint to register the wizard bot webhook. The super admin calls it once after deployment. The endpoint is idempotent — calling it again overwrites the existing webhook registration.

This endpoint follows the auth guard pattern from `(admin)/layout.tsx` (SUPER_ADMIN_EMAIL check) but as an API route instead of a layout guard.
